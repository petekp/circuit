## Strengths
- It correctly identifies that command metadata is currently threaded through multiple layers: manifest fields in `schemas/circuit-manifest.schema.json:45-52`, extraction in `scripts/runtime/engine/src/catalog/extract.ts:66-76`, rendering in `scripts/runtime/engine/src/cli/catalog-compiler.ts:39-49`, and install verification in `scripts/verify-install.sh:208-229`.
- It also correctly calls out that `workers` is not currently misclassified as a circuit. The extractor emits `kind: "utility"` whenever `circuit.yaml` is absent (`scripts/runtime/engine/src/catalog/extract.ts:49-88`), and `ARCHITECTURE.md` explicitly describes `workers` as an adapter utility, not a circuit (`ARCHITECTURE.md:41-43`, `ARCHITECTURE.md:497-501`, `ARCHITECTURE.md:870-883`).

## Weaknesses
- The proposal creates the exact ownership problem it claims to remove. Today the compiler discovers workflows straight from the filesystem and manifests: `extract()` walks `skills/*` and emits catalog entries from what exists on disk (`scripts/runtime/engine/src/catalog/extract.ts:42-88`), and `catalog-compiler.ts` renders the circuit table from that extracted catalog (`scripts/runtime/engine/src/cli/catalog-compiler.ts:39-60`). Proposal A replaces that with a manual join on `id` (`proposal-a.md:87-92`, `proposal-a.md:201-205`). That means a contributor can add `skills/new-workflow/circuit.yaml` and still fail to surface it unless they also remember a second file. That is not “one owner.” That is a second checklist.
- Its “registry is the only owner of slash-command identity” claim is false on its face. Slash-command identity already lives in `SKILL.md` frontmatter triggers and direct-invocation prose, and the proposal does not scope removing that. Examples: `skills/build/SKILL.md:8-25`, `skills/explore/SKILL.md:10-27`, `skills/review/SKILL.md:7-8`, `skills/handoff/SKILL.md:6-8` and `skills/handoff/SKILL.md:24-33`, `skills/workers/SKILL.md:3-5`, and `skills/run/SKILL.md:6-10` plus `skills/run/SKILL.md:19-27`. Moving aliases out of `circuit.yaml` does not make the registry the sole owner; it just adds another place that can disagree with the skill docs.
- The “role” move is a half-measure that makes the type system worse. `scripts/runtime/engine/src/catalog/types.ts:1-4` says the catalog types are the single contract between extractor, generator, and validator, and no other module may define its own catalog shape. Proposal A then introduces semantic role as important enough to justify a new registry, but explicitly refuses to put it into that shared contract (`proposal-a.md:195-197`, `proposal-a.md:302-305`). That guarantees shadow types and repeated join logic. If role matters, it belongs in the shared type system. If it does not belong in the shared type system, it is not stable enough to justify a new source file.
- It does not eliminate mirror fields. It moves them into a bigger mirror. For circuits, the repo already enforces the convention that `expert_command === /circuit:<id>` (`scripts/runtime/engine/src/catalog/catalog-validator.test.ts:79-87`). The plugin namespace is also already treated as a stable invariant across docs (`scripts/runtime/engine/src/release-integrity.test.ts:903-956`). Proposal A then writes out `alias: /circuit:<id>` line-by-line in `commands/registry.yaml` for every command. That is not removing duplication. That is serializing a convention into a new file and calling it architecture.
- Its command model is already lossy. The proposed schema reduces command invocation to `task_arg: required|forbidden` (`proposal-a.md:44-84`). But `handoff` already supports `/circuit:handoff done` (`skills/handoff/SKILL.md:6-8`, `skills/handoff/SKILL.md:30-33`), and the docs explicitly mention direct workflow overrides like `/circuit:build --rigor deep <task>` (`skills/run/references/rigor-profiles.md:118`). So the registry’s core abstraction cannot even faithfully represent the current surface. That is not a small omission. It means the model is wrong.
- The proposal misframes `workers` as mere accidental leakage. The shipped `workers` skill explicitly advertises `/circuit:workers` in its own frontmatter (`skills/workers/SKILL.md:3-5`), and there is a matching shim at `commands/workers.md:1-5`. Deleting `commands/workers.md` is therefore a product decision that revokes a documented invocation, not a harmless cleanup. Proposal A treats that as if it were just removing dead duplication. It is not.
- The “minimal change” story around generation is fantasy. `catalog-compiler.ts` currently patches exactly two existing marker blocks in `CIRCUITS.md` (`scripts/runtime/engine/src/cli/catalog-compiler.ts:62-74`). `generate.ts` only patches inline marker blocks in existing files; it does not create files, delete files, or manage directories (`scripts/runtime/engine/src/catalog/generate.ts:14-35`, `scripts/runtime/engine/src/catalog/generate.ts:37-69`). Meanwhile the shims are standalone files (`commands/build.md:1-5`, `commands/run.md:1-5`, `commands/workers.md:1-5`), the utilities section in `CIRCUITS.md` is hand-authored (`CIRCUITS.md:23-28`), and the command tables in `README.md` are hand-authored (`README.md:75-105`). Making the registry authoritative means adding a file emitter, deletion logic, and a broader docs-generation strategy. That is not “tiny sibling emitter” work.
- The `verify-install.sh` split is not a simplification. The current script already validates the shipped plugin surface from a script-relative `PLUGIN_ROOT` and is explicitly designed to run from repo or installed cache (`scripts/verify-install.sh:1-4`). It checks hooks, schemas, skills, shims, relay scripts, and bundled CLIs, but does not read `README.md`, `CIRCUITS.md`, or `ARCHITECTURE.md` at all. Proposal A’s `repo|installed|auto` split (`proposal-a.md:211-213`, `proposal-a.md:229-260`) exists because the registry introduces a new “freshness vs shipped artifact” distinction. That is differently complex, not simpler.

## Hidden Assumptions
- It assumes contributors will reliably update `commands/registry.yaml` whenever they add or rename a workflow. Current discovery is filesystem-driven (`scripts/runtime/engine/src/catalog/extract.ts:42-88`). Proposal A replaces that with a second manual step and does not acknowledge the resulting failure mode.
- It assumes `commands/registry.yaml` is safe to place inside `commands/`. Today that directory is treated as a shim directory: verification checks `commands/<skill>.md` (`scripts/verify-install.sh:215-225`), sync copies it as command shims (`scripts/sync-to-cache.sh:64-70`), and the sync tests assert the installed layout contains `commands/run.md` (`scripts/runtime/engine/src/sync-to-cache.test.ts:172-191`). There is no evidence anywhere in the repo that non-Markdown control files belong there.
- It assumes shim descriptions are derivable without adding another override field. That is not true in the current repo. `commands/run.md` uses the short description “The primary Circuit router.” (`commands/run.md:1-5`), while `skills/run/SKILL.md` and `skills/run/circuit.yaml` carry materially longer text (`skills/run/SKILL.md:3-8`, `skills/run/circuit.yaml:5-8`). `commands/handoff.md` is also a compressed summary of a much richer frontmatter description (`commands/handoff.md:1-5`, `skills/handoff/SKILL.md:3-8`). So either the generated shims get worse, or the registry grows another “short description” field and reintroduces duplication.
- It assumes docs can stop being tested without becoming stale. But `README.md` and `CIRCUITS.md` are still hand-authored user-facing command surfaces (`README.md:75-105`, `CIRCUITS.md:23-28`). Deleting command-surface tests without making those sections generated does not produce cleaner ownership. It produces undocumented drift.
- It assumes making `workers` non-public is an obvious correctness fix. The codebase does not support that certainty. Architecture docs call it an adapter (`ARCHITECTURE.md:41-43`, `ARCHITECTURE.md:497-501`), but the skill itself still advertises `/circuit:workers` (`skills/workers/SKILL.md:3-5`). That is an unresolved product choice, not a schema invariant.

## Feasibility
It is implementable in the narrow sense that the repo already has YAML parsing (`scripts/runtime/engine/src/catalog/extract.ts:6-8`) and JSON-Schema validation helpers (`scripts/runtime/engine/src/schema.ts:37-72`). But it is not implementable as casually as the proposal claims. The hard part is not loading one more YAML file. The hard part is redefining the public command contract without leaving `SKILL.md`, `README.md`, `CIRCUITS.md`, shims, and verification policy disagreeing with each other.

The scope is materially larger than advertised. To make this real, someone has to add a new schema and loader, define extracted-catalog-to-registry completeness invariants, create file create/delete generation for `commands/*.md`, decide whether `README.md` and the Utilities section of `CIRCUITS.md` become generated or merely unverified prose, migrate existing slash-command references in skill docs, and resolve the fact that the proposed command model cannot represent current surfaces like `handoff done`. That is not a cleanup patch. That is a control-plane rewrite in miniature.

## Verdict
FATALLY FLAWED

Proposal A diagnoses a real mess, but its cure is worse: it moves command identity into a new manually maintained registry, leaves semantic role outside the shared type contract, fails to model the current command surface accurately, and understates the generator and verification work needed to stop everything else from drifting. It does not eliminate duplication. It redistributes it and adds a new place to forget.
