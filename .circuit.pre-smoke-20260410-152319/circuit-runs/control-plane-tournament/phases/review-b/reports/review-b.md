## Strengths

- Proposal B correctly preserves the one part of the current catalog system that is actually working: the `extract -> generate` spine is small and deterministic today (`scripts/runtime/engine/src/catalog/extract.ts:30-91`, `scripts/runtime/engine/src/catalog/generate.ts:14-69`, `scripts/runtime/engine/src/cli/catalog-compiler.ts:77-117`).
- It correctly identifies one real semantic gap: the shared catalog type only distinguishes `"circuit"` vs `"utility"`, while the architecture docs distinguish workflows, lifecycle utilities, and the `workers` adapter (`scripts/runtime/engine/src/catalog/types.ts:6-27`, `ARCHITECTURE.md:41-43`, `ARCHITECTURE.md:497-500`, `ARCHITECTURE.md:591-618`).
- It is also correct to reject the false claim that `workers` is currently extracted as a circuit. In this repo it is extracted as a utility, and the tests prove it (`scripts/runtime/engine/src/catalog/extract.ts:79-87`, `scripts/runtime/engine/src/catalog/extract.test.ts:108-121`).
- It correctly notices that public slash-command status is currently inferred from directory presence. `verify-install.sh` walks every `skills/*` directory and fails if `commands/<skill>.md` is missing (`scripts/verify-install.sh:215-225`).

## Weaknesses

- This proposal is aimed at the wrong layer. The real runtime kernel does not run on the catalog at all; it runs directly on `circuit.manifest.yaml`. `derive-state()` loads the manifest and projects runtime state from `manifest.circuit.*` (`scripts/runtime/engine/src/derive-state.ts:27-35`, `scripts/runtime/engine/src/derive-state.ts:61-83`). `resume.ts` builds the step graph and start step straight from the manifest (`scripts/runtime/engine/src/resume.ts:107-124`). By contrast, the catalog compiler only has two subcommands, `generate` and `catalog`, and only patches `CIRCUITS.md` or emits JSON (`scripts/runtime/engine/src/cli/catalog-compiler.ts:6-8`, `scripts/runtime/engine/src/cli/catalog-compiler.ts:80-103`). So `KernelIR` is not a kernel redesign. It is a second model next to the real kernel.
- The proposal says it is eliminating mirror fields, then immediately creates a much larger mirror. `WorkflowRuntime` duplicates `version`, `purpose`, `signals`, `entryModes`, `steps`, `executor`, `kind`, `protocol`, `writes`, `gateKind`, and `routes` even though `ARCHITECTURE.md` already defines `circuit.yaml` as the machine-readable topology (`ARCHITECTURE.md:24-35`, `ARCHITECTURE.md:706-735`). The actual manifests already carry exactly that topology (`skills/run/circuit.yaml:1-38`, `skills/build/circuit.yaml:1-145`, `skills/explore/circuit.yaml:1-111`). Replacing two command fields with a lossy copy of the entire manifest is not anti-mirror design. It is a bigger mirror.
- Embedding `role`, `visibility`, and especially `aliases` into `SKILL.md` frontmatter creates a brand new authoring-vs-runtime coupling. Today the extractor’s frontmatter contract is intentionally weak: it parses the YAML block, types it as `Record<string, string>`, and only reads `name` and `description` (`scripts/runtime/engine/src/catalog/extract.ts:16-27`, `scripts/runtime/engine/src/catalog/extract.ts:46-47`, `scripts/runtime/engine/src/catalog/extract.ts:75-76`, `scripts/runtime/engine/src/catalog/extract.ts:84-85`). Proposal B’s sample frontmatter requires enums and an array, which the current extractor does not validate or even type honestly. Worse, `ARCHITECTURE.md` defines `SKILL.md` as the execution contract that the orchestrator follows line by line (`ARCHITECTURE.md:24-39`). Turning that markdown file into the authority for install visibility and slash aliases is exactly the kind of authoring/runtime coupling the proposal claims to avoid.
- There is already evidence that SKILL frontmatter is loose author metadata, not a durable runtime contract. `skills/review/SKILL.md` frontmatter includes `trigger`, which the extractor ignores completely (`skills/review/SKILL.md:1-9`, `scripts/runtime/engine/src/catalog/extract.ts:75-76`, `scripts/runtime/engine/src/catalog/extract.ts:84-85`). Proposal B wants to take that same loose frontmatter channel and make it govern public surface and role classification. That is a trust upgrade with no schema or validator behind it.
- The proposal is wildly over-engineered for the actual governance problems in this repo. The current pain points are narrow and concrete: utilities are collapsed into one bucket (`scripts/runtime/engine/src/catalog/types.ts:6-27`), `workers` is missing from the manual utility table in `CIRCUITS.md` even though it ships a command shim (`CIRCUITS.md:23-29`, `commands/workers.md:1-5`), and `verify-install.sh` currently forces every skill directory public (`scripts/verify-install.sh:215-225`). None of that requires `KernelIR`, `kernel-ir.json`, a shipped-surface manifest, new CLI projection modes, and a verifier split. A small registry or a minor extension of the current catalog would solve the real problems directly.
- The shipped-surface manifest is bureaucracy without demonstrated payoff. Current cache sync already has a clear root allowlist and explicit sync behavior (`scripts/sync-to-cache.sh:26-34`, `scripts/sync-to-cache.sh:55-89`). The test suite locks that six-root cache layout today (`scripts/runtime/engine/src/sync-to-cache.test.ts:183-217`). Proposal B’s exact-file manifest collides with the fact that marketplace sync intentionally leaves extra files alone and commits synced changes in place (`scripts/sync-to-cache.sh:93-113`, `scripts/runtime/engine/src/sync-to-cache.test.ts:195-217`). Either marketplace extras become false-positive drift, or the manifest is not actually the “sole install-surface contract.”
- The manifest also adds maintenance burden for every nested shipped file, not just top-level roots. The repo currently ships nested helper files under `skills/handoff/scripts/`, `skills/run/references/`, and `skills/workers/references/`. Proposal B turns that into a per-file release inventory problem even though the concrete issues visible in this checkout are much smaller: accidental publicness in `verify-install.sh` and a missing `workers` row in the manual `CIRCUITS.md` utilities table (`scripts/verify-install.sh:215-225`, `CIRCUITS.md:23-29`, `commands/workers.md:1-5`). That is a huge increase in release bookkeeping for a tiny plugin surface.
- The “No test parses prose” invariant is actively bad. The prose tests exist because user-facing docs are part of the product contract. `lifecycle-regressions.test.ts` checks bootstrap and transfer guidance in SKILL docs and `CIRCUITS.md` prose tables (`scripts/runtime/engine/src/lifecycle-regressions.test.ts:83-145`, `scripts/runtime/engine/src/lifecycle-regressions.test.ts:307-348`). `release-integrity.test.ts` checks README command docs, utilities documentation, and command-shim existence (`scripts/runtime/engine/src/release-integrity.test.ts:153-179`, `scripts/runtime/engine/src/release-integrity.test.ts:845-850`, `scripts/runtime/engine/src/release-integrity.test.ts:961-995`). Replacing those with IR-only tests would stop verifying whether humans are being told the truth.

## Hidden Assumptions

- Proposal B assumes private SKILL-based units are an actual requirement. This repo currently ships no such thing. Every skill directory has a corresponding command shim, and both verification and release-integrity tests assume that is true (`scripts/verify-install.sh:215-225`, `scripts/runtime/engine/src/release-integrity.test.ts:961-995`).
- It assumes migrating away from `entry.command` and `expert_command` is local catalog work. It is not. Those fields are embedded in the manifest schema (`schemas/circuit-manifest.schema.json:45-68`), in round-trip fixtures used by `verify-install.sh` (`scripts/verify-install.sh:78-96`), in runtime CLI integration fixtures (`scripts/runtime/engine/src/runtime-cli-integration.test.ts:40-57`), in architecture docs (`ARCHITECTURE.md:706-727`), and in all six workflow manifests under `skills/*/circuit.yaml`.
- It assumes verifier splitting is mostly rearrangement. But `scripts/verify-repo.sh` does not exist in this repo today. That is a new tool to invent, not a file to tweak. Proposal B understates that scope completely.
- It assumes a generated manifest meaningfully replaces existing ownership checks. It does not address the fact that release integrity currently spans `README.md`, `CIRCUITS.md`, `docs/workflow-matrix.md`, `ARCHITECTURE.md`, SKILL docs, hooks, manifests, and command files (`scripts/runtime/engine/src/release-integrity.test.ts:845-920`, `scripts/runtime/engine/src/release-integrity.test.ts:923-1010`, `scripts/runtime/engine/src/lifecycle-regressions.test.ts:67-110`, `scripts/runtime/engine/src/lifecycle-regressions.test.ts:307-348`). Most of that contract is intentionally human-facing prose.

## Feasibility

- The proposal understates how many files really move. Beyond the catalog folder, changing alias ownership requires updating `schemas/circuit-manifest.schema.json` (`schemas/circuit-manifest.schema.json:45-68`), the schema regression that validates all manifests (`scripts/runtime/engine/src/schema-regressions.test.ts:278-310`), `verify-install.sh` round-trip fixtures (`scripts/verify-install.sh:78-96`), runtime CLI integration fixtures (`scripts/runtime/engine/src/runtime-cli-integration.test.ts:40-57`), six checked-in workflow manifests, and the checked-in built CLI artifact `scripts/runtime/bin/catalog-compiler.js`.
- The proposal explicitly budgets rewrites for catalog tests plus `release-integrity.test.ts`, `lifecycle-regressions.test.ts`, and `sync-to-cache.test.ts`, but even that list proves the migration is not “preserve the compiler spine, swap the IR.” The current catalog shape is referenced in at least eight engine source/test files already (`scripts/runtime/engine/src/catalog/types.ts`, `scripts/runtime/engine/src/catalog/extract.ts`, `scripts/runtime/engine/src/catalog/generate.ts`, `scripts/runtime/engine/src/cli/catalog-compiler.ts`, `scripts/runtime/engine/src/catalog/extract.test.ts`, `scripts/runtime/engine/src/catalog/generate.test.ts`, `scripts/runtime/engine/src/catalog/catalog-validator.test.ts`, `scripts/runtime/engine/src/lifecycle-regressions.test.ts`), and the doc/command surface is enforced by three more high-friction test files (`scripts/runtime/engine/src/catalog/catalog-validator.test.ts`, `scripts/runtime/engine/src/lifecycle-regressions.test.ts`, `scripts/runtime/engine/src/release-integrity.test.ts`).
- Even if all of that work lands, the runtime still will not consume `KernelIR`. The live engine continues to consume manifests directly (`scripts/runtime/engine/src/derive-state.ts:27-35`, `scripts/runtime/engine/src/derive-state.ts:61-83`, `scripts/runtime/engine/src/resume.ts:107-124`). So the proposal demands a large migration while failing to delete the existing runtime source of truth. That is the opposite of durable.
- The rewrite scope around tests is especially unrealistic because many tests are not accidental string scraping. They intentionally compare manifest topology to human docs, command files, and SKILL narrative guidance (`scripts/runtime/engine/src/release-integrity.test.ts:35-91`, `scripts/runtime/engine/src/release-integrity.test.ts:579-601`, `scripts/runtime/engine/src/release-integrity.test.ts:633-767`, `scripts/runtime/engine/src/lifecycle-regressions.test.ts:149-199`, `scripts/runtime/engine/src/lifecycle-regressions.test.ts:307-348`). Proposal B treats that as disposable noise. It is actually the contract surface.
- The simplest useful fixes are much smaller than Proposal B admits. Stop deriving publicness from directory presence, generate the utilities section in `CIRCUITS.md`, and add a tiny explicit registry for public utility roles. Those changes attack the actual bugs in `CIRCUITS.md` and `verify-install.sh` directly (`CIRCUITS.md:23-29`, `scripts/verify-install.sh:215-225`) without turning the catalog into a second manifest system.

## Verdict

FATALLY FLAWED

Proposal B correctly spots a few real paper cuts, but its cure is to bolt a second control-plane description language onto a runtime that already has one. The result is not a stronger kernel. It is a larger mirror, a new markdown/runtime coupling, a heavier release process, and a test rewrite that throws away intentional doc-contract coverage while leaving the real manifest-driven kernel in place.
