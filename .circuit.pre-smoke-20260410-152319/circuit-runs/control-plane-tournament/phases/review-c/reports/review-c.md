## Strengths
- Proposal C correctly identifies a real drift: `workers` is classified as a `utility` because `extract.ts` treats any skill without `circuit.yaml` that way (`scripts/runtime/engine/src/catalog/extract.ts:49-88`), and `ARCHITECTURE.md` describes `workers` as a utility/adapter rather than a workflow (`ARCHITECTURE.md:41-43`, `ARCHITECTURE.md:583-585`, `ARCHITECTURE.md:870-883`). But `commands/workers.md` exists (`commands/workers.md:1-5`), and `verify-install.sh` currently requires a command shim for every `skills/*` directory (`scripts/verify-install.sh:215-225`). That is a real inconsistency.
- It is also right that the shipped install surface is operationally encoded today in `scripts/sync-to-cache.sh`, not in a declarative manifest. The script hard-prunes cache targets to `.claude-plugin`, `commands`, `hooks`, `schemas`, `scripts`, and `skills` (`scripts/sync-to-cache.sh:21-35`) and then rsyncs those trees explicitly (`scripts/sync-to-cache.sh:55-91`).

## Weaknesses
- This is not a command-identity solution. It is a packaging solution pretending to be a kernel solution. Proposal C admits it "does not fully solve command identity" (`.circuit/circuit-runs/control-plane-tournament/phases/diverge-c/reports/proposal-c.md:75-76`), and the existing code shows that gap is fatal, not cosmetic. `extract.ts` still binds identity across directory name, `circuit.id`, `SKILL.md` frontmatter, and command strings (`scripts/runtime/engine/src/catalog/extract.ts:66-85`). `catalog-validator.test.ts` hard-requires `dir === id`, `SKILL.md name === id`, and `expert_command === /circuit:<id>` (`scripts/runtime/engine/src/catalog/catalog-validator.test.ts:62-87`). `ARCHITECTURE.md` reinforces the same rule: `circuit.id` "matches directory name" and `expert_command` "matches /circuit:<id>" (`ARCHITECTURE.md:711-718`). None of that changes if you add `shipped-surface.yaml`. You have fixed who gets copied, not what a command is.
- The proposal creates a second metadata kernel in direct contradiction to the existing one. `catalog/types.ts` opens with: "This is the single contract between extractor, generator, and validator. No other module may define its own catalog shape." (`scripts/runtime/engine/src/catalog/types.ts:2-4`). Proposal C's step 2 is to add a separate loader/validator pair under `scripts/runtime/engine/src/shipped-surface/` (`.circuit/circuit-runs/control-plane-tournament/phases/diverge-c/reports/proposal-c.md:85-86`). That is not simplification. It is duplicate control-plane metadata by design.
- It completely punts on the one place where the current surface is already non-trivial: `entryCommand` versus `expertCommand`. `run` has both fields today (`skills/run/circuit.yaml:10-12`). `extract.ts` preserves both (`scripts/runtime/engine/src/catalog/extract.ts:72-74`). `catalog-compiler.ts` renders the public invocation from `entryCommand` when present, otherwise `expertCommand` (`scripts/runtime/engine/src/cli/catalog-compiler.ts:43-48`). Proposal C introduces `public.slash_aliases` but never says what wins if that alias differs from `entry.command` or `expert_command`, or which one should be rendered into `CIRCUITS.md`. That is not a minor omission. `run` is the primary public entrypoint of the plugin (`CIRCUITS.md:3`, `CIRCUITS.md:19`, `README.md:8`, `README.md:81-88`).
- `shipped-surface.yaml` is not "one owner." It is another mirror. Public command identity already leaks into multiple authored surfaces besides `commands/`. `skills/run/SKILL.md` says "Also accessible as /circuit:run" (`skills/run/SKILL.md:4-8`). `skills/build/SKILL.md` says "Use for /circuit:build" (`skills/build/SKILL.md:8-10`). `skills/workers/SKILL.md` says `Use for "/circuit:workers"` (`skills/workers/SKILL.md:4-6`). `README.md` hardcodes the public workflows and utilities (`README.md:94-105`). `CIRCUITS.md` does too (`CIRCUITS.md:15-28`). Moving one copy of export state into YAML does not make export state singular.
- The manifest sketch already misses currently required shipped bytes. That is a bad sign for a proposal whose entire selling point is package precision. The sample manifest includes `scripts/runtime/bin` and `schemas` (`.circuit/circuit-runs/control-plane-tournament/phases/diverge-c/reports/proposal-c.md:45-49`), but the current installed surface also requires `scripts/relay/compose-prompt.sh`, `dispatch.sh`, and `update-batch.sh` (`scripts/verify-install.sh:252-292`), `circuit.config.example.yaml` (`scripts/verify-install.sh:234-249`), and `scripts/verify-install.sh` itself plus bundled CLIs in integration tests (`scripts/runtime/engine/src/runtime-cli-integration.test.ts:79-89`). Proposal C is trying to replace an implicit package contract with an explicit one, yet its own sketch does not enumerate the actual contract.
- It hand-waves away the hardest part of `sync-to-cache.sh`: delete semantics. The current script does not sync arbitrary roots. It prunes cache targets by top-level directory name (`scripts/sync-to-cache.sh:21-35`), then runs `rsync --delete` against whole `skills/`, `commands/`, `scripts/`, and `schemas/` subtrees (`scripts/sync-to-cache.sh:61-91`). Tests lock in the resulting cache shape: exactly `.claude-plugin`, `commands`, `hooks`, `schemas`, `scripts`, and `skills` (`scripts/runtime/engine/src/sync-to-cache.test.ts:183-191`). Marketplace extras are intentionally preserved (`scripts/runtime/engine/src/sync-to-cache.test.ts:195-218`). Inference from those sources: a nested-root manifest like `skills/build` or `scripts/runtime/bin` requires a brand-new deletion algorithm for stale siblings under shared parents. If the helper does not recreate the current whole-tree `--delete` behavior exactly, stale installed files survive. If it does, you have rewritten `sync-to-cache.sh` in Node and added YAML on top.
- Proposal C's "No test parses prose" invariant is actively hostile to this codebase's architecture. `ARCHITECTURE.md` says `SKILL.md` is the execution contract and that `circuit.yaml` and `SKILL.md` "must agree" (`ARCHITECTURE.md:24-39`). `release-integrity.test.ts` is explicitly designed to guard "drift between docs, manifests, skills, and tests" (`scripts/runtime/engine/src/release-integrity.test.ts:1-6`) and deliberately parses headings and artifact templates from prose (`scripts/runtime/engine/src/release-integrity.test.ts:35-91`, `scripts/runtime/engine/src/release-integrity.test.ts:73-91`, `scripts/runtime/engine/src/release-integrity.test.ts:633-699`). You can reduce markdown scraping for package checks. You cannot honestly claim this system should stop parsing prose when prose is part of the executable spec.

## Hidden Assumptions
- The proposal assumes the root problem is packaging drift. The current code says otherwise: identity is still hard-wired in `extract.ts`, `catalog-validator.test.ts`, and `ARCHITECTURE.md` (`scripts/runtime/engine/src/catalog/extract.ts:66-85`, `scripts/runtime/engine/src/catalog/catalog-validator.test.ts:62-87`, `ARCHITECTURE.md:711-718`).
- It assumes publicness can be made singular by moving it into `shipped-surface.yaml`. But command names are already duplicated in SKILL frontmatter, README, CIRCUITS, and the generated catalog-facing docs (`skills/run/SKILL.md:4-10`, `skills/build/SKILL.md:8-10`, `README.md:94-105`, `CIRCUITS.md:15-28`).
- It assumes the hidden `/circuit:workers` surface is just verifier fallout. But `commands/workers.md` exists (`commands/workers.md:1-5`), `verify-install.sh` requires it (`scripts/verify-install.sh:215-225`), and `skills/workers/SKILL.md` advertises it (`skills/workers/SKILL.md:4-6`). That is not an incidental leak. It is a deliberately shipped command today, even if the docs choose not to mention it.
- It assumes package metadata can live cleanly outside `catalog/`. The current generator, validator, and docs already depend on the catalog contract (`scripts/runtime/engine/src/cli/catalog-compiler.ts:15-18`, `scripts/runtime/engine/src/catalog/generate.ts:14-34`, `scripts/runtime/engine/src/catalog/catalog-validator.test.ts:90-115`). Splitting shipping metadata into a separate subtree does not remove coupling. It just moves the drift boundary.

## Feasibility
- A manifest-driven packaging hardening pass is implementable. Proposal C as written is not a small hardening pass. To match current behavior it would need to rework `sync-to-cache.sh`, `verify-install.sh`, `catalog-compiler.ts`, generated docs, command shims, release-integrity expectations, sync tests, and runtime-install integration tests (`scripts/sync-to-cache.sh:21-113`, `scripts/verify-install.sh:190-320`, `scripts/runtime/engine/src/cli/catalog-compiler.ts:39-74`, `scripts/runtime/engine/src/runtime-cli-integration.test.ts:79-102`, `scripts/runtime/engine/src/sync-to-cache.test.ts:152-280`, `scripts/runtime/engine/src/release-integrity.test.ts:958-997`).
- If the actual goal is command-identity decoupling, the proposal is not feasible as described because it explicitly leaves the runtime identity rules untouched (`.circuit/circuit-runs/control-plane-tournament/phases/diverge-c/reports/proposal-c.md:75-77`) while the live validator still enforces them (`scripts/runtime/engine/src/catalog/catalog-validator.test.ts:62-87`). You cannot claim "runtime unit identity != slash-command alias" while leaving the current identity invariants intact.
- If the actual goal is only package hygiene, the separate `shipped-surface/` loader/validator is over-engineered. The repo already has one shared catalog contract (`scripts/runtime/engine/src/catalog/types.ts:2-4`) and one shipping script with explicit tests (`scripts/runtime/engine/src/sync-to-cache.test.ts:194-280`). A second metadata subsystem is harder than the proposal admits and buys less than it claims.
- Inference from the current files: the clean version of this idea would be a narrow packaging fix to stop `verify-install.sh` from treating every skill as public. Proposal C instead turns that into a new control-plane layer without resolving the pre-existing identity knot. That is high implementation cost for partial benefit.

## Verdict
FATALLY FLAWED

It diagnoses a real symptom, but it attacks the packaging layer while leaving the live command-identity kernel intact. Worse, it does that by adding a second metadata system next to the one file that explicitly says there should only be one.
